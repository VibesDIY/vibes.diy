# Implementing the codegen auto-roll: the admission gate is one wrapper, the hard part was where to count

Source: PR #2829, branch `claude/chat-shard-cold-start-7clrke`. The design (sibling seed `2026-06-29-chat-shard-cold-start.md`) called for per-user codegen shard pinning + admission-control auto-roll. Implementing it surfaced two decisions the spec only gestured at.

**Where to count concurrency.** The "active stream" unit is an in-flight `prompt-chat-section` generation — and that handler's promise stays unresolved for the whole generation. So admission control is just a wrapper around that one handler: `if (active >= limit) send shard-overloaded; else active++ / try { run } finally { active-- }`. The `finally` is load-bearing — it's what frees the slot on a mid-stream WS close, not just clean completion. The counter lives on the Durable Object instance (shared by reference across concurrent requests) and is injected into the AppContext exactly like the existing `shardIdentity` gate — its presence is the signal "enforce here." The subtle correctness win: inject it **only for per-user (notify-prefixed) shards**, so random-UUID CLI/anon DOs get zero admission control and stay byte-for-byte behavior-preserving. The limit isn't "never hit until clients pin" — it isn't even installed.

**Why the client roll can't be a transparent retry.** The spec said "the pending request is retried on the new shard." But a prompt's section stream is bound to the WebSocket connection at `LLMChat.create` time — the `onMessage` subscription and the stream writer both capture that `conn`. If `VibesDiyApi.request` transparently re-sent the prompt after rolling, the generation would stream on the *new* connection while `LLMChat` is still listening to the dead one. So the roll instead **advances the shard index, persists it (sessionStorage, so reloads stay on the working shard), and drops the connection** — which lets the *existing* reconnect/reopen machinery re-establish the stream on the fresh shard. The roll reuses infrastructure that already exists for disconnects rather than reinventing streaming-replay. Same trick that made it cheap: the live socket URL is rebuilt from `base + codegenShardForUser(uid, index)` on every `getReadyConnection`, so a roll takes effect without re-keying the provider's instance cache.

The cache key was the bug Codex caught pre-implementation and it stayed true in code: keying the lazy `chatApi` by the bare `/api` URL means the pre-Clerk anon instance wins forever. Keying it by `${apiUrl}#codegen-user:${uid}` is what lets the post-hydration render actually move to the per-user shard. One line, but the whole feature is inert without it.
