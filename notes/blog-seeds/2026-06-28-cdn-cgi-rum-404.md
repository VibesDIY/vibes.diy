# The 404 your own Worker route is causing: /cdn-cgi/rum

Source: `vibes.diy/pkg/workers/route-decision.ts`, `vibes.diy/pkg/workers/app.ts`

Every page load fired `POST /cdn-cgi/rum` → 404, a console + failed-network entry on
every route (#2770). Cloudflare auto-injects its Web Analytics beacon
(`static.cloudflareinsights.com/beacon.min.js`), which POSTs telemetry to
`/cdn-cgi/rum`. Benign, but it pollutes the console during QA and buries real errors.

The non-obvious part worth a post:

- **`/cdn-cgi/*` is normally edge-handled — but a zone-wide Worker route shadows it.**
  The intuition "`/cdn-cgi/` is Cloudflare's, not mine" is *usually* right, so the
  first instinct is to chase a zone setting (enable RUM, or turn off auto-injection).
  But our Worker owns the whole `vibes.diy/*` route, so the beacon POST lands in our
  `fetch` handler, not Cloudflare's collector. With no matching route it falls through
  to React Router SSR, which 404s. The fix is therefore *in code*, not in the dashboard:
  a one-line route case that returns `204 No Content`.

- **204, not a proxy.** We could try to forward the beacon to the real collector, but
  `fetch(request)` from inside the Worker goes to origin (which has no `/cdn-cgi/rum`
  handler either), so there's nothing to proxy to. RUM data was already being lost to
  the 404; a 204 just makes the loss *quiet*. If we ever want the analytics back, that's
  the zone-config path — out of scope for silencing the noise.

Gotcha: scope the match to the exact path `=== "/cdn-cgi/rum"`, not `startsWith`.
`/cdn-cgi/trace` is a legitimate edge endpoint and other `/cdn-cgi/*` paths shouldn't be
swallowed — only the beacon collector we're actively shadowing.
