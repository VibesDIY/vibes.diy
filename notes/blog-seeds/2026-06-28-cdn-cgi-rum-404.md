# The Cloudflare RUM beacon 404 you can't fix in your Worker (until you can)

Source: `vibes.diy/pkg/workers/app.ts`, `vibes.diy/pkg/app/root.tsx`, `vibes.diy/pkg/app/vibes-diy-provider.tsx`

Every page load fired `POST vibes.diy/cdn-cgi/rum` → 404 — a console + failed-network
entry on every route (#2770). Cloudflare's Web Analytics "Automatic Setup" injects its
RUM beacon (`static.cloudflareinsights.com/beacon.min.js`) and, for ad-blocker evasion,
points it at your **own zone** path `/cdn-cgi/rum`. The whole arc is worth a post because
the obvious fixes are all wrong:

- **You can't intercept it in the Worker.** First instinct: add a route that returns 204
  for `/cdn-cgi/rum`. Dead end — `/cdn-cgi/*` is served by Cloudflare's *edge* before the
  request reaches any Worker. Proof: `curl /cdn-cgi/trace` returns Cloudflare's trace dump
  (`fl=…`, `colo=…`) that no Worker code generates, and the `/cdn-cgi/rum` 404 carries
  `server: cloudflare` + `cf-ray` + CF's `nel`/`report-to` headers, not our SSR's. A
  Worker route for it is dead code that never executes. (We tried; we deleted it.)

- **Automatic injection is unreliable for Worker-rendered HTML.** vibes.diy is 100%
  SSR-in-the-Worker, and Cloudflare's edge HTML-rewriter injection + same-origin collector
  don't reliably work when the Worker owns the response — which is exactly why the
  zone-path beacon 404s here.

- **The fix flips to manual setup — and *that* is in-repo code.** Switch the dashboard to
  "Enable with JS Snippet installation" and inject the beacon yourself in SSR `<head>`.
  The manually-installed beacon posts to `cloudflareinsights.com/cdn-cgi/rum` (Cloudflare's
  own domain), so it never touches our zone or Worker — no 404, same dashboard.

The gotcha that ties back to GDPR: manual setup loses Cloudflare's automatic **"exclude
EU"** carve-out. But because we render in the Worker we have `request.cf.isEUCountry` — so
we rebuild the exact same posture in one line (`enableCfRum = env.ENVIRONMENT === "prod" &&
request.cf?.isEUCountry !== "1"`) and only emit the (cookieless) beacon for non-EU
visitors. The `=== "prod"` half matters: `import.meta.env.DEV` only excludes the Vite dev
server, but deployed dev/preview/cli Workers are real `react-router build` outputs — and
because a manual-install beacon attributes data by *token*, not hostname, a prod token
firing from a preview host pollutes the production analytics site. Token is a public
client-side value, hardcoded in `root.tsx`; the env/EU gate is the only per-request logic. Trade-off accepted: the `cloudflareinsights.com` collector is
more ad-blockable than the same-origin zone path, which is a fine price for killing the
404 on every page.
