# Verifying an auth ship from a headless cloud box: the localhost callback the agent proxy eats

Source: prod verification of the de-fireproof identity ship (`ship@2.7.1`, the keybag lift from #2716 / the `2026-06-28-keybag-lift` seed), branch `claude/verify-identity-ship-prod-ld9z0n`. Wrote up the runbook as `agents/identity-ship-verify.md`.

The ship's whole contract is "nothing on the wire changed â€” no forced re-login." So the verification isn't "does it build", it's "do *existing* credentials still authenticate end-to-end." Three checks: browser sign-in (Clerk decode + lifted dash-client), a headless device-id round-trip (seed the prod cert via `$VIBES_DEVICE_ID`, `list`, expect a personalized listing â€” proves keybag writeâ†’readâ†’ES256 signâ†’worker verify), and the one the seed can't fake: a real interactive `vibes-diy login` that drives the CSRâ†’CAâ†’persist enroll path.

That third check is where the time went, and the two gotchas are the reusable part:

- **The agent proxy eats the localhost callback.** `vibes-diy login` stands up a local HTTP server on a random port and waits for the signed cert to be POSTed back to `http://localhost:<port>/cert`. Drive the authed browser (launched with the mandatory cloud `proxy: { server: HTTPS_PROXY }` for the HTTPS `vibes.diy` leg) to the CA page, and the cert *is* signed â€” but the browser routes the `http://localhost` redirect **through the agent proxy too**, which only accepts HTTPS CONNECT tunnels and answers with `agent-proxy relay: this proxy only accepts HTTPS CONNECT tunnels`. The cert never reaches the CLI. `proxy.bypass: "localhost,127.0.0.1"` did **not** help. What works: `context.route()` to intercept any `localhost`/`127.0.0.1` request and service it with a direct Node `http.get` (no proxy), then `route.fulfill()` the response back â€” the Node GET delivers the cert to the CLI's local server and enrollment completes.

- **Don't close the browser before the 5s redirect timer.** The `/settings/csr-to-cert` page auto-submits only after Clerk reports `isLoaded && isSignedIn`, then waits a hardcoded `waitUntilClose = 5s` before redirecting to the callback. Clerk init + that timer is â‰ˆ 8â€“12s. A driver that screenshots-and-closes at ~8s (the obvious default) kills the page mid-flight and `login` times out. Wait until the main-frame URL actually becomes `localhost:<port>`.

The wider lesson worth a post: a "verify the deploy" task in a sandboxed cloud agent has an extra failure axis beyond the app â€” the **execution environment's network policy**. The same proxy that makes outbound HTTPS work silently breaks an inbound-to-localhost OAuth-style callback, and the symptom (login hangs, cert never arrives) looks like an app bug, not a proxy bug. Worth its own section in any "drive a real auth flow headlessly" doc.

Non-secret hygiene that belongs in the muscle memory: never echo `$VIBES_DEVICE_ID` or the storage-state cookie, redact `csr=`/`cert=` in captured output, and `rm -rf` every scratch `HOME` (they hold device-cert keybags). The interactive check also enrolls a *new real* device cert server-side â€” inherent to testing the enroll path, not a leak, but worth stating so the next verifier isn't surprised.
