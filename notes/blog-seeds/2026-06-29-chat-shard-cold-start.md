# Trading cold-start latency for CPU isolation — and then earning the isolation back

Source: branch `claude/chat-shard-cold-start-7clrke`, design spec `docs/superpowers/specs/2026-06-29-chat-shard-cold-start-design.md`. The website's heavy codegen chat opens a fresh `crypto.randomUUID()` Durable Object shard on every session (`api/impl/index.ts:290`), which means a cold DO and a 10–15s wait every time. The random sharding wasn't an accident — the comment at `index.ts:284` says it plainly: "Each API instance gets its own DO shard to avoid CPU limits under concurrent load." Random UUIDs are a load-balancer that buys CPU isolation at the cost of guaranteed cold starts.

The fix everyone reaches for first — "pin each user to one DO" — is the easy half. The hard half is that **pinning re-creates the exact problem the random sharding solved**: now one user's tabs all land on one DO, and a burst of concurrent generations can exhaust it. So the design isn't "stop random-sharding," it's "stop random-sharding *and* re-earn the isolation a different way." That second way is admission control: the DO admits at most 3 concurrent codegen streams, and the 4th gets a deterministic `shard-overloaded` signal instead of a crash.

The non-obvious decision is **admission control over crash-detection.** Cloudflare CPU/memory exhaustion kills the isolate, often before it can send a clean error — so you can't reliably catch it and react. Counting active streams and rejecting *before* the heavy work starts turns "hope we survive the limit" into a deterministic, testable signal. The client then rolls to the next shard in a per-user family (`notify-user-<uid>` → `notify-user-<uid>~1`) and retries.

The gotcha that makes the family non-trivial: there's an anti-forgery guard at `session-callbacks.ts:88` that ties a user's shard to their verified `userId` with **exact string equality** (`shard !== userNotifyShardFor(userId)`). Roll to `~1` and that guard silently no-ops notify registration — the rolled shard isn't "the" user shard anymore. So the roll forces a guard generalization from "is this *the* shard" to "does this shard *belong to* this user's family," preserving the security property (you still can't register someone else's shard) while admitting the rolls. And the CLI gets all of this for free by doing nothing: it never passes a `shardKey`, so it stays on random UUIDs and never rolls. The feature is opt-in through a seam that already existed.
