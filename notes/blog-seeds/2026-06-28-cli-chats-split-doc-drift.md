# When a CLI command splits in two, the docs that named it go stale silently

`vibes-diy chats` did one fuzzy thing — show "the chats" — so it got split into two sharp commands: `codegen-log` (the builder↔LLM build transcript) and `app-chats` (the deployed app's own runtime chat/image data). The split is good design — the old single verb conflated two genuinely different surfaces (how the app was *built* vs. what the app *stored*). But the rename left live documentation pointing at a command that now just prints a migration notice: the `vibe-data` and `vibe-code` skills both still walked users through `vibes-diy chats ...`.

The gotcha: a removed-to-stub command doesn't *fail loudly* — it exits 0 with a "use these instead" message — so doc drift here is invisible until someone copy-pastes a dead example. Caught it validating `ship@2.6.11`: the agents/ runbook had already been updated in the same PR that did the split, but the two **skills** hadn't. Lesson worth a post: when you retire a CLI verb, grep the whole repo for the bare command string (`vibes-diy <verb>`) across `agents/`, `.claude/skills/`, and READMEs in the same PR — the stub's politeness is exactly what hides the rot. (The trade-off we kept: leave the stub printing a pointer rather than removing it outright, so old muscle-memory and old docs both self-correct instead of erroring.)
