# Pulling the trigger on a wire-token contract: the producers were ready, the test suite wasn't

Source: #2618 contract step, branch `claude/issue-2618-status-sriszb`. The third and final beat of an expand→migrate→contract rollout that renamed the on-the-wire session `mode` from `chat`/`app` to `codegen`/`runtime`. #2630 widened the wire to accept both token families; #2641 flipped every in-tree producer to emit canonical; this PR drops legacy acceptance entirely.

The deferral logic from #2608 was sound: `mode` round-trips to clients in a way catalog/settings tags don't — `openChat` echoes it back, and old published CLIs gate `dryRun`/`focusPath`/`selected` on `mode === "chat"`. Normalize the echo too early and `generate --dry-run` silently starts creating real vibes. So the rule was: don't remove legacy acceptance until old CLI builds have aged out. "Pull the trigger" = that day arrived.

The gotcha that made this bigger than a three-line type flip: **"flip the producers" and "remove acceptance" are different-sized jobs, and the gap between them is the test suite.** #2641 dutifully flipped the *product* producers — CLI, dashboard, sandbox. But the api test suite opens sessions with `api.openChat({ mode: "chat" })` in ~40 places as a convenient default, and those are producers too. They kept compiling only because the wire still *accepted* `chat`. The moment you make `PromptLLMStyle` canonical-only, every one of those is a type error. Removing acceptance isn't "delete two union arms" — it's "migrate every in-tree caller, including the ones that were never about back-compat."

The type system turned that from scary to mechanical: narrow the wire type, run `tsc`, and it enumerates every stale `mode: "chat"` for you. A blind `sed` over the enumerated test files finished the job — the only sites that *keep* legacy tokens are the ones asserting they're now **rejected** (`chat-types.test.ts` flipped from "accepts legacy + canonical" to "canonical-only, legacy rejected").

The subtlety worth preserving: **not all `chat`/`app` acceptance is the wire.** The model *catalog* `ModelCapability` still accepts the legacy tags — a persisted `AIParams.model` embeds a `Model` whose `supports` tags may have been written before #2608, and that data-at-rest back-compat is independent of the wire `mode` and outlives it. So `canonicalModelUsage()` / `LEGACY_CAPABILITY_ALIASES` stay (they normalize stored catalog tags, and defensively normalize the raw pre-validation `req.mode` at the shard-routing layer), while the now-dead wire bridges in the validated handlers — `canonicalModelUsage(req.mode)` where `req.mode` is provably canonical-only — collapse to direct comparisons. Removing legacy and keeping legacy in the same PR, on two layers that happened to share a vocabulary.
